🤖 Running the fleet · Part 09 of 10

The proving ground: the dev mesh

Before a change reaches a space that a real person uses, it is proven on a second, smaller fleet that is sealed off from production by construction: a two-box, two-country mesh that holds no member data, physically cannot reach the production object store, and updates from its own release feed. It exists so that the production fleet can stay hard to touch while the software that runs on it can still be changed quickly and safely.

You are here: the whole map again, but as a second, complete copy of itself — its own peer boxes, its own object store, and its own release feed. Nothing on this page shares state with the production fleet; that separation is the point.

Read this first: what the dev mesh is, and is not

The dev mesh is a build tool, not a place any member’s space lives. It carries no real chats, no real accounts, and no production credential of any kind. Because it is sealed off from production and holds nothing worth stealing, it is deliberately relaxed and disposable — the opposite of the hardened production boxes. Everything on this page describes that second fleet, which by design runs ahead of production on in-review builds, so a “running today” badge here means “operable on the dev fleet today”, not “already in the production release channel”.

Why a separate environment exists

The production fleet is deliberately difficult to touch. Operator access is over a private network tunnel only; direct login as the all-powerful root user is switched off; the few privileged actions an operator may take are fixed in a reviewed allowlist rather than a shell; and every operator session is written to a public ledger. That posture is exactly right for machines that hold members’ conversations — and it makes iteration slow. Standing up a new piece of infrastructure, watching it behave, and tearing it down again is a careful, audited exercise every time.

The wrong way to make iteration faster would be to loosen production. The right way is a second fleet that is safe to be relaxed, precisely because it is walled off from everything that matters. That is the dev mesh: a place to stand things up and tear them down without the security ceremony, while still being a real, multi-box mesh — so nothing built there can quietly assume a single machine, and everything reaches production the disciplined way (a signed release, self-installed), never a hand-edit on a live box.

In plain words

Production is the locked building where the real conversations are kept, and getting in requires signing the book every time. The dev mesh is an identical but empty practice building next door: same floor plan, same locks, but nothing valuable inside, so you can rearrange the furniture freely. You only move a change into the real building once it has worked in the practice one.

What it is: two boxes, two countries, two providers

The dev mesh is the smallest fleet that proves the distribution model rather than merely asserting it: two boxes, in two different countries, on two independent hosting providers, running on two different CPU architectures. One box would prove nothing about a mesh; two boxes that differ in every dimension that production will eventually differ in prove a great deal.

The country labels (ie, fi) are mechanism-exercise only; because the dev mesh holds no member data, they carry no data-residency obligation, unlike the same labels in production (see the production model below).

Sealed off from production — by construction, not by convention

“Cut off from production” is not a rule someone remembers to follow; it is a property of how the dev mesh is built. Three separations make it structural:

  • A different object store. Mesh membership is nothing more than heartbeat objects written under a prefix in a shared bucket. The dev mesh uses its own, separate bucket, so a dev box physically cannot join the production mesh — not even if it were misconfigured to try. The same bucket also holds the dev mesh’s hibernation data and its per-space leases, so no dev process ever reaches for a production object at all.
  • A different release feed. Production boxes follow the signed production channels (part 04). The dev boxes are meant to follow a dev feed instead, so that a dev publish can never surprise a production box and a production publish never disturbs dev. Promotion from dev to production is always a separate, deliberate publish — never an accident of a shared channel.
  • Test-only secrets. Any credential on a dev box is a sandbox or test credential — a test AI key, test payment keys — never a production one. There is no member data to protect and no production system a leaked dev secret could reach.
Why sealed-by-construction rather than sealed-by-configuration

A separation that depends on someone setting the right value can be undone by someone setting the wrong one. A separation that is baked into the shape of the system — a different bucket, so the coordinating storage room is simply a different room — cannot be undone by a typo. The strongest form of “these two things cannot touch” is when the mechanism that would let them touch does not exist between them.

Naming: a parallel universe

Production’s per-country scheme places a space at <space>.<country>.11ch.at, rooted at the bare product domain. The dev mesh is a parallel universe rooted one level deeper, at dev.11ch.at, with the identical country structure beneath it:

Putting the environment outermost — dev as the widest label, with the country nested inside it — means every dev name in the whole fleet sits under one structural root. That is what makes “everything dev” a property of the name tree itself, not merely of configuration: one wildcard per country-and-environment points at the box that serves it, and a dev name can never be mistaken for, or resolve as, a production one.

What the dev mesh proves

Two things, one about correctness and one about process.

Per-country placement is correct

The mesh (part 07) decides which box owns a space by hashing the hostname over the live boxes. On a single-country fleet that is enough. On a fleet where different boxes serve different countries, plain hashing is not enough: the hash could name a box in one country the owner of a space that belongs to another country — a box that then cannot serve it, because it holds no certificate for that country’s domain and no routing row for that host. Ownership and ability would disagree, and the space would be stranded.

The fix, which the two-country dev mesh is exactly the rig to prove, is to scope placement to the boxes that can actually serve a host. Each box advertises which country-domains it covers; both ownership (the hash) and the replication of routing rows are then narrowed to the boxes whose domains include the host in question. A space in one country is therefore owned, woken, and served only by that country’s box, and its routing row is never even copied to the other country’s box. Proving this needs two boxes that genuinely disagree about who the raw hash would pick — which is precisely what two boxes in two countries give you. On a single-wildcard production mesh (two boxes both serving the same domain) the scoping is a harmless no-op, so the same code is safe everywhere.

Changes roll out the disciplined way

The load-bearing rule the dev mesh enforces is simple: everything ships by publishing to a feed and letting boxes install it themselves; nothing ships by editing a live box by hand. A change is published to the dev feed, both dev nodes are watched as they converge on it, and it is exercised across the real two-box topology — and only then is it published to production. The dev mesh is where single-box assumptions and rollout breakage are found, which is the whole reason it is a mesh and not a single box. If a change cannot be rolled out cleanly to two boxes that differ in country, provider, and architecture, that is a fact worth learning before members are involved, not after.

The production model this rehearses

The dev mesh is not an end in itself; it is a rehearsal for a production endgame that is mostly still design. In that endgame, every country is its own subdomain with a wildcard pointing at that country’s mesh, so routing splits into a coarse, almost-never-changing part (which country, decided in DNS) and a fine, dynamic part (which box in that country owns the space, decided by the in-country lease and hash). A space in a given country stays in that country, with no cross-country proxy hop, and data residency becomes a property of the space’s name — a public promise the infrastructure must be able to keep before any country subdomain ships. The same per-country substrate is where a co-operative of independent providers eventually plugs in: a country’s wildcard could point at any enrolled in-country provider’s mesh.

Designed, not built: the per-country production fleet

None of the production per-country model above is operable today. Production runs a small number of boxes that share a single wildcard domain, so the placement scoping this dev fleet proves is currently a no-op there. Country subdomains, the residency promise that would ride on them, per-country object stores as authoritative primaries, and the provider co-op seam are all written design (docs/planet-scale.md, docs/dev-mesh.md) with schema fields or interfaces cut for them, and no member-facing behaviour yet. The dev mesh exists so that when those do ship, they ship having already run.
