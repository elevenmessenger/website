🤖 Running the fleet · Part 01 of 10

Overview and orientation

This guide explains, from the ground up, how the Eleven Messenger fleet is deployed and operated: what runs where, how each piece is started and supervised, and exactly which commands you type to add, remove, inspect, update, and repair things.

The whole system on one map. Every detailed page in this guide reproduces this map with the part it covers highlighted, so you always know where you are.

Who this guide is for, and what it assumes

This guide is written for a person who needs to operate this system — not for the people who wrote it. It assumes you are comfortable with the basics of system administration: you can open a terminal, connect to a server with ssh, edit a file, and run a command. It assumes only a passing familiarity with systemd (the Linux service manager), DNS, and object storage — every one of those is explained where it first matters, and again in the glossary.

Everything here is deliberately repetitive in places. Key ideas — for example, that one chat space is one process with one database file, or that the object-storage lock is the real source of truth about who may write — are restated wherever they matter, because internalising those few ideas makes every command in this guide predictable rather than memorised.

In plain words

You are looking after a small number of Linux servers. Each server runs one program, called lchat, which plays several roles at once. Chat groups live on these servers as small, separate processes, one per group. Idle groups are put to sleep and their data is parked in Amazon’s object storage. This guide teaches you how to look after all of that.

What the product is, in one paragraph

Eleven Messenger is an end-to-end encrypted group chat product. Every chat group — called a space — runs as its own fully isolated server-side unit: its own process, its own SQLite database file, its own admin token, and its own subdomain (for example chat-a.example.com). The server side only ever stores and relays ciphertext — encrypted bytes it cannot read — because all encryption and decryption happens on members’ devices. This matters operationally: nothing you do as an operator can expose message content, because the servers never hold the keys. Copying a database file, replicating it to object storage, or moving it between machines is safe by construction.

The one idea behind “planet scale”

The system is designed so that it can grow to a very large number of spaces without a large number of servers, and the design is captured in the repository document docs/planet-scale.md. The core idea has three parts:

  • One space, one process, one database file — forever. Spaces are never consolidated into a shared database. This is simultaneously the isolation story (one space crashing cannot hurt another), the sharding story (a space can move to any machine by moving one file), and the security story.
  • An idle space does not deserve a running process. Because a space’s entire state is one SQLite file of ciphertext, that file can be parked in object storage (Amazon S3) when nobody is talking, and the process can exit. When someone next opens the space, the system restores the file and starts the process again, in a few seconds. This is called hibernation.
  • Several servers can share the work without a central coordinator. Each server (this guide calls them boxes) announces itself by writing a small heartbeat object into the shared S3 bucket. A well-known hashing rule decides which box should host which space, and a per-space lock object in S3 guarantees that two boxes can never write the same space’s data at the same time. This is called the mesh.

The vocabulary this guide uses everywhere

Nine words carry most of the weight in this guide. They are defined properly in the glossary, but you will get much further if you internalise them now:

WordMeaning
spaceOne chat group, as members experience it. Server-side, a space is one instance (next row).
instanceThe server-side unit behind a space: one process, one SQLite database file, one subdomain. This guide uses “instance” when talking about servers and “space” when talking about the product; they name the same thing from two sides.
hostAn instance’s full domain name, for example chat-a.example.com. The host is the identity of an instance: it names its database file, its systemd unit, its socket, and its certificate.
boxOne Linux server in the fleet. A box runs the router, many instances, and the supporting services.
the routerThe one process on a box that listens on the public ports (80 and 443). It holds the TLS certificates and forwards each request to the right instance. See part 02.
the registryA small SQLite database (/data/registry.db) on each box listing every host the box serves and how to reach it. The router reads its routing table from here.
unitA systemd service definition. Instances run under a template unit called lchat@<host>.service; the router runs under lchat-router.service. See part 03.
the bucketThe Amazon S3 bucket used for hibernation: parked space databases, per-space lock objects, and mesh heartbeats all live here. See part 06.
the leaseThe per-space lock object in the bucket. Whichever process holds a space’s lease is the only process allowed to write that space’s data. The lease is the ultimate tie-breaker in the whole design.

One program, many roles

Almost everything in this system is a single, statically compiled Go program called lchat. There is no interpreter to install, no runtime to patch, and no package manager involved: deploying software to a box means copying one file. The same binary behaves differently depending on the subcommand it is started with:

InvocationRoleExplained in

To the router they are just two more names in the routing table; they are covered in part 03.

Following one request through the system

To make the map concrete, here is what happens when a member opens https://chat-a.example.com in a browser:

  1. DNS resolves chat-a.example.com to a box’s public address. (A wildcard DNS record usually covers every subdomain at once, so new spaces need no DNS work.)
  2. The router on that box accepts the connection on port 443 and presents the TLS certificate for that hostname — either a per-host certificate it obtained automatically from Let’s Encrypt, or a wildcard certificate.
  3. The router looks the hostname up in the registry and finds the instance’s unix socket, /run/lchat/chat-a.example.com.sock.
  4. If the instance is awake, the router forwards the request over that socket and the instance answers. If the instance is hibernating, the router’s built-in activator first claims the space’s lease in S3, restores the database if needed, starts the instance process as a child, and then forwards the request (this adds a few seconds, once).
  5. If this box is part of a mesh and the hashing rule says a different box owns this space, the router transparently forwards the whole request to that box instead.

Every step of that journey has its own page in this guide, and each page walks the same journey from its own component’s point of view.

How to read this guide

The parts are ordered so that each one only depends on the ones before it. If you are new to the system, read parts 02 and 03 before anything else — they establish the router and the process model that everything else builds on. If you are here to do a specific task, part 08 is a task-oriented runbook with step-by-step procedures, each of which links back to the explanatory pages.

  1. Overview and orientation — this page.
  2. The router — the front door of every box: TLS certificates, the routing table, the registry.
  3. Everything that runs on a box — each process, its systemd unit, how it is started, supervised, configured, and inspected; the filesystem layout.
  4. Installing and updating the software — the version store, staged deploys, health checks, rollback, release channels, the hourly auto-updater, and pinning.
  5. Creating and removing chat spaces — add-instance and remove by hand; the control daemon, the cloud provisioner, quotas, and the reaper.
  6. Hibernation — how an idle space’s process exits and its data parks in object storage, and how a request (or a schedule) wakes it again.
  7. The mesh — how several boxes share one fleet: heartbeats, placement by hashing, proxying, and draining a box.
  8. Day-to-day operations — the runbook: checking what is running, reading logs, health checks, backups and restore, migrating an instance between boxes.
  9. The proving ground — the dev mesh: a sealed-off, two-country practice fleet where every change is proven before it reaches production.
  10. Complete reference — every command with every flag, every environment variable, every file path, every systemd unit, and the glossary.

Conventions used throughout

  • Command blocks say where to run them. Every terminal example carries a label such as “Run on the box, as root” or “Run on your own machine”. Lines beginning with # inside a block are explanatory comments, not commands.
  • Placeholders are angle-bracketed. <host> always means an instance’s full domain name, such as chat-a.example.com. <ver> means a build version identifier (a short git commit hash). Replace them, brackets and all.
  • Example names are consistent. The guide uses chat-a.example.com as its running example instance and box-1 / box-2 as example servers, so you can follow the same imaginary fleet across pages.
  • “In plain words” panels restate the essential idea of a section without any terminology, on purpose, so you can check your understanding as you go.
  • “Why it is this way” panels explain a design decision where knowing the reason makes the behaviour easier to remember.
