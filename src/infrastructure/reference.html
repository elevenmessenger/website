Command reference: the lchat control commands

All of these run on a box (over ssh), as root, invoking /opt/lchat/lchat. Common conventions: <host> is always a full domain name; flags may come before or after the host; most commands accept --registry <path> to point at a non-default registry (you will essentially never need it — the default is /data/registry.db).

Box and instance lifecycle

lchat bootstrap
One-time box setup: writes every systemd unit file, creates the lchat system user and the /data, /run/lchat, and certificate-cache directories, mints the root-only control token, enables and starts the router and the hourly update timer. Idempotent — safe to re-run to refresh unit files. Explained in part 04.
lchat add-instance <host> [--admin-token <tok>] [--hibernate] [--app-store-url <url>] [--home-origin <url>] [--push-relay-url <url>] [--push-relay-token <tok>]
Creates a space: mints (or, with --admin-token, imports) the admin token, writes /data/<host>.env, registers the route, points the version link, starts and health-checks the unit (skipped with --hibernate), reloads the router, warms the certificate, and prints the one-time setup link and the admin link. Explained in part 05.
lchat add-service <host> --socket <path>
Registers a service-kind route (home server, push relay, cloud provisioner, or any third-party backend) after verifying the socket answers. No unit is created and no version is tracked — the router only forwards. Explained in part 03.
lchat add-bot <host> [--model <model>]
Provisions the bot sidecar for an instance: mints one agent invite per identity, writes /data/<host>-bot.env and the session directory, and enables lchat-bot@<host> (on a hibernating host the unit is skipped; the activator co-spawns instead). Reads ANTHROPIC_API_KEY from the calling environment if AI behaviour is wanted. Explained in part 03.
lchat remove <host> [--stop]
Deletes the host’s registry row and reloads the router (traffic stops); with --stop also stops and disables the instance and sidecar units. Never touches data. Explained in part 05.
lchat hibernate <host> --on | --off
Moves one host between systemd management and activator management, in either direction, safely ordered so the host is never unserveable. Requires the router to be running with a hibernation bucket for --on. Explained in part 06.
lchat replication <host> [stream | batched | daily]
Reads (no argument) or sets a hibernating host’s replication class — how often awake writes sync to the bucket. A set reloads the router and applies at the host’s next wake; on provisioned fleets billing drives this instead. Explained in part 06.
lchat instances
Prints the registry: HOST, KIND, STATUS, VERSION, HIBERNATE, REPLICATION, SOCKET, plus a last error line for any failed row. The first stop for “what does this box think it is serving?”.
lchat mesh-seed <peer-url>
Run on an existing mesh box, pointed at a newly joined one: bulk-replicates every in-scope hibernating route to the peer in one pass, authenticated by LCHAT_MESH_TOKEN. The one-time catch-up step of adding a box to the mesh. Part 07.
lchat admin-link <host> / lchat setup-invite <host>
Recover the admin link (from the environment file — the only plaintext copy of the token) / mint a fresh one-time setup invite.

Versions and deploys

lchat stage --file <binary> [--version <ver>]
Copies a built binary into the version store under its build identifier (asked of the binary itself unless overridden). Inert and always safe. Part 04.
lchat deploy <host>|--all [--version <ver>] [--pin] [--include-pinned]
With --version: repoint the host’s live link, restart, poll /api/version for the exact build, auto-roll-back on mismatch, restart the bot sidecar. Without: restart on the current build. --all rolls one host at a time, stops on first failure, and skips pinned hosts unless --include-pinned. --pin (single host + version only) freezes the host there. Hibernating hosts: pure repoint, effective at next wake. Part 04.
lchat rollback <host> | --all
Repoints to the recorded previous version, restarts, health-checks, and swaps current/previous (so rollback twice returns forward). Part 04.
lchat init-versions [--from <binary>]
One-time adoption when moving an existing box onto versioned deploys: stages a seed build and creates missing live links. Idempotent; never repoints an existing link.
lchat update run [--dry-run] / status / pin <host> [<ver>] / unpin <host>
One updater pass (what the hourly timer fires) / the fleet update table (HOST, CHANNEL, AUTO, CURRENT, AVAILABLE) / freeze a host on a version / resume its channel. Part 04.
lchat version
Prints the binary’s own build identifier (also served by every instance at GET /api/version as build).

Daemons and utilities

lchat router
The router process itself — run by lchat-router.service, not by hand. Flags --registry, --acme-cache, --https, --http, --fallback-url, --wildcard-domains. Part 02.
lchat controld
The root-only control daemon — run by lchat-control.service. Flags/env for socket, token file, and capacity policy are under environment variables. Part 05.
lchat cloud --socket <path> --db <path>
The self-serve provisioner — run by lchat-cloud.service. Part 05.
lchat bot run [--base <url>] / lchat bot provision
Run a bot sidecar (with no arguments, configuration comes from the unit’s environment file) / claim an agent invite into a session file. Part 03.
lchat release keygen|sign|verify|payload
The release-feed signing plumbing — used by the publishing pipeline, not in box operation. Part 04.

Glossary

space / instance / host
One chat group; its server-side unit (one process + one SQLite database + one subdomain); and that subdomain itself, which names everything belonging to the instance. See the vocabulary table.
box
One Linux server in the fleet, running the router and everything in part 03.
systemd
The standard Linux service manager: it starts processes at boot, restarts them when they crash, applies sandboxing, and keeps their logs (the “journal”). Controlled with systemctl, read with journalctl.
unit / template unit
A systemd service definition; a template unit (name containing @) is stamped out per instance name, which is how one file defines the service for every chat space.
unix socket
A file (here under /run/lchat/) that local processes connect to like a network address. Invisible to the network — the reason instances need no firewall rules of their own.
the registry
The SQLite file /data/registry.db: one row per hostname a box serves. The router’s routing table and the fleet commands’ ledger. Part 02.
DNS / wildcard DNS
The system translating names to network addresses. A wildcard record (*.example.com) points every subdomain at once, which is why creating a space needs no DNS step.
Let’s Encrypt / ACME / HTTP-01 / DNS-01
The free certificate authority and its automated issuance protocol. HTTP-01 proves control of a hostname by answering a challenge on port 80 (used per host); DNS-01 proves control of a whole domain via a temporary DNS record (used for wildcard certificates). Part 02.
object storage / S3 / bucket / prefix
Storage addressed as named objects in a “bucket” rather than files on a disk — durable, replicated, and shared. A “prefix” is the leading part of object names, used like a directory. Amazon S3 is the implementation used here.
Litestream
A tool that continuously streams a SQLite database’s changes into object storage, allowing restore to any recent moment. Used twice in this system, in two forms: a stock copy for fleet backups, and embedded inside the router for hibernation.
the lease
The per-space lock object (lock.json) in the hibernation bucket, held while a process serves the space and renewed every few minutes. The system’s single source of truth about who may write a space’s data. Part 06.
replication class
A hibernating host’s durability rung (stream / batched / daily): how often its awake writes sync to the bucket, set per plan by billing or per host by lchat replication. Graceful sleeps and deploys always flush in full regardless. Part 06.
the activator
The component inside the router that wakes sleeping spaces on demand and puts idle ones to sleep. Exists only when LCHAT_HIBERNATE_BUCKET is set. Part 06.
rendezvous hashing
The rule mesh boxes use to agree which of them owns each space without communicating: every box scores every (box, space) pair the same way, and the highest score wins. Adding or removing a box only moves the spaces whose winner changed. Part 07.
bot sidecar
The separate process hosting an instance’s bot members. It holds the bots’ encryption keys (which must never live in the server process) and connects outward as an ordinary client. Part 03.
canary / pin
Deploying a new build to one instance to watch it before promoting it everywhere; and the mark that freezes an instance on a version so the hourly auto-updater leaves the experiment alone. Part 04.
web push / VAPID key
Browser push notifications, signed per instance with a key pair (the .vapid file). Distinct from Apple device push, which goes through the push relay.
