The routine health check, in ninety seconds

For the fleet-wide view without logging into boxes, the public status dashboard (below) publishes instance counts, awake/asleep split, capacity bands, version spread, and replication health every five minutes.

Reading the logs: the lines worth knowing

Everything logs to the systemd journal. The single most informative journal is the router’s, because the activator and mesh live inside it (part 06):

Log line (pattern)MeaningAction
<host> waking / awake / hibernatedThe activator’s normal lifecycle.None — this is the system working.
LEASE LOST … standing downThis box could not renew a space’s lease; it stopped serving the space, which will re-wake on the next request (possibly on another box, in a mesh).Occasional: fine. Frequent: investigate bucket reachability and credentials.

Backups: continuous replication, and the restore you should rehearse

Every database on the box — instance databases, the registry, the home, push, and cloud databases — is continuously replicated to a dedicated backup bucket by a stock Litestream process (lchat-litestream.service, configuration /etc/litestream.yml). This is replication as a property of normal operation, not nightly snapshots: recovery-point loss is measured in seconds, and point-in-time restore is available. Three properties of the arrangement matter operationally:

  • The configuration is generated, not written. deploy/fleet-backup.sh regenerates /etc/litestream.yml from what is actually in /data; run it after adding instances so new databases get picked up. It deliberately skips hibernating hosts — their replication belongs to the activator, and two replicators on one database corrupt each other (part 06).
  • The bucket is the ransomware posture. It keeps old versions of every object for fourteen days, and the box’s credentials can write objects but can never change versioning, lifecycle, or the bucket itself — so even an attacker with the box’s own keys cannot destroy the history they overwrite.
  • Blobs and the certificate cache ride separately. Attachment files (/data/<host>-blobs/) are immutable and content-addressed; the certificate cache (/data/acme/) is key material. Both belong in whatever file-level backup wraps the box.

Restoring a database (rehearse this before you need it)

  1. Always restore to a scratch path — never over a live file.

  2. Only after verifying, swap it into place with the instance stopped.

For a hibernating host the parked copy in the hibernation bucket is usually the freshest state; a cold wake restores it automatically. The backup bucket is your answer when the hibernation copy itself is what was damaged.

Moving an instance between boxes

Migration is data movement plus a DNS change — nothing about members’ keys or history is affected, provided the hostname never changes (browser encryption keys and passkeys are bound to the origin). The procedure, condensed from the repository’s validated runbook (docs/instance-migration.md):

  1. Recover the admin token and per-instance configuration from the source box.

    Both live in /data/<host>.env. The token must be preserved — importing it on the destination (rather than minting fresh) is what keeps every saved admin link working.

  2. Snapshot the database cleanly, on the live system.

    VACUUM INTO produces a single, transaction-consistent, compacted copy and is safe against a running instance — no stop required for the snapshot itself.

  3. Copy three things, not one.

    The snapshot to /data/<host>.db on the destination; the blob directory (rsync -a /data/<host>-blobs/ … — immutable files, safe to copy live); and the web-push key file /data/<host>.vapid, without which every browser push subscription silently breaks.

  4. Import on the destination, preserving the token.

    add-instance adopts the pre-placed database, starts the unit against it, health-checks, and reloads the router (part 05). Verify over the local socket before touching DNS.

  5. Check the gap, then flip DNS.

    Compare message counts between source-now and the snapshot; if the space is busy, freeze writes and re-snapshot at cutover. Lower the DNS record’s TTL to about sixty seconds a while before the flip. The TLS certificate cannot be pre-issued (issuance validates over public DNS), so it issues on the first request after the flip — a few seconds, once.

  6. Verify, and keep the source as the instant rollback.

    Leave the source instance running until you are satisfied: flipping DNS back is a complete rollback. Bot sidecars are their own step — their identity files (/data/<host>-bot/) must be copied, never re-provisioned with add-bot (that would mint duplicate bot members).

The hardening checklist (one-time per box)

  • Firewall: allow inbound 80, 443, and the WireGuard port (51820, UDP) only; default-deny the rest. SSH gets no public allow — it is permitted only on the WireGuard interface (ufw allow in on wg0 to any port 22 proto tcp). Everything internal is unix sockets, so nothing else needs a port. (Add the allows before enabling the firewall, and verify the tunnel from outside before removing any public SSH rule, so you do not lock out your own session.) running today
  • SSH rides WireGuard: each operator is a peer on the box’s wg0 interface and connects to the box’s tunnel address; the tunnel runs persistently on the operator’s machine with an ssh configuration alias, so every existing command works unchanged. WireGuard does not answer unauthenticated packets, so the SSH port is invisible to scanners and brute-force noise drops to zero. Key-only remains in force (PasswordAuthentication no). The full recipe, including the break-glass path through the hosting provider’s rescue console, is in docs/instance-router.md under Hardening. running today
  • Root SSH is bootstrap-only: the first root session bootstraps the box and creates named operator accounts (group lchat-operators), installs the committed sudo allowlist (deploy/sudoers-operators → /etc/sudoers.d/lchat-operators) and the lchat-env-install helper for placing secrets, then sets PermitRootLogin no. From then on operators are named users whose root capability is exactly the allowlist — anything new that needs root gets an allowlist line through a reviewed pull request, never a root shell. This is accident-proofing and attribution with a reviewable surface, not a hard security boundary (the allowlist can deploy binaries); the hard boundary is the WireGuard key. running today
  • Automatic security updates for the operating system (unattended-upgrades) — the application updates itself (part 04), but the kernel and OpenSSL do not.
  • Pin the box’s own hostnames to loopback in /etc/hosts, so on-box clients (bot sidecars, calls to the push relay) reach the local router instead of the public internet — and disable any cloud-image behaviour that regenerates that file on boot. The subtle failure this prevents is described in part 03.
  • The unit files already sandbox every service (part 03) — nothing to do, but worth knowing when something cannot write where you expected: ProtectSystem=strict means the whole filesystem is read-only to services except /data and /run/lchat.

What you can observe, and what you cannot yet

A status publisher on one box regenerates a set of public dashboards every five minutes, including a fleet page whose data file (fleet.json) reports: total, systemd-managed, and hibernating instance counts; how many hibernators are awake versus asleep fleet-wide at this moment; the software version spread; up/down booleans for the sibling services; backup-replication health with its lag in seconds; and a coarse capacity band. The capacity band — plenty, getting-full, or at-capacity — is the worst of several internal signals (instance count against the cap, registered rows against the row ceiling, disk against the floor, memory), and deliberately only the band word is published: never counts of slots, gigabytes, percentages, hostnames, or addresses. The same discipline runs through everything public: aggregate numbers and band words, nothing that identifies a machine or a member.

In a mesh, the dashboard aggregates across every live box, not just the one running the publisher. Each mesh member — including that one — publishes its own capacity snapshot as a sibling object next to its membership heartbeat, on a slower (one-minute) cadence; the publisher reads every live peer’s snapshot back over the same S3 substrate the mesh already uses, no new box-to-box port involved. Each box’s own reported awake/asleep count is filtered to the hosts it currently owns by the mesh’s placement hash — so a host that has been re-homed to a peer is never reported as “asleep” by more than one box at once, and the fleet page’s hosts array lists every box exactly once, each with its own whimsical, hash-derived label (never a hostname or address) and, optionally, an operator-set coarse region (LCHAT_BOX_REGION, “EU · Finland”-shaped — continent and country only).

Designed, not built: real metrics

Wake latency, lease conflicts, and replication lag exist today as log lines, not as queryable metrics with alerting. The design owes an end-to-end-encryption-safe metrics discipline; until it lands, trend questions (“are wakes getting slower?”) are answered by reading journals, and the five-minute dashboard is the only aggregation. Budget your expectations accordingly.

lchat instances shows a host as failedIts creation or last restart did not become healthy in time.Read the row’s last error line, then the instance’s journal. The files are left in place on failure; fixing the cause and re-running the operation is safe.
A hibernating host’s first wake fails with a permission errorIts environment file is root-owned (the activator reads it unprivileged).chown lchat:lchat /data/<host>.env — and note the flip commands do this themselves now; hand-copied files are the remaining way to hit it. See part 06.
A burst of 503 responses on one hostWake contention: a lease held elsewhere, or a wake exceeding its budget.Check the router journal for lease messages. In a mesh, confirm both boxes see the same membership (list the heartbeats). Occasional 503s during wakes are by design; sustained ones are not.
A bot stops answering after a migration or DNS changeThe sidecar dials the public hostname, and the box resolves it to the wrong place.Verify the /etc/hosts loopback pin on the box that runs the sidecar; restart the sidecar unit. See part 03.
deploy or update refuses: “another deploy… is running”The deploy lock is held.Usually a genuinely running hourly pass — check systemctl status lchat-update.service; a timer pass that lost the lock simply retries next hour.
