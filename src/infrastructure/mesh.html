The mesh: running more than one box

The mesh lets several boxes serve one fleet of hibernating spaces with no coordinator, no shared database, and no new network protocol between them: boxes announce themselves with heartbeat objects in the shared S3 bucket, a hashing rule decides which box should own which space, and the per-space lease guarantees correctness even when boxes disagree.

You are here: the peer boxes, and the bucket they coordinate through. Note that peers never need to discover each other directly — everything flows through the bucket they already share for hibernation.

Read this first: the mesh is live

Production runs two boxes today, both with LCHAT_MESH_SELF set, splitting a shared pool of hibernating spaces under a scoped set of domains (LCHAT_MESH_DOMAINS). Registry rows now replicate automatically between them (see route replication below) — the manual step this section used to describe is gone. The honest limits section at the end lists what genuinely remains unbuilt; everything else on this page describes code as it runs today.

The idea: ownership as a rule, correctness as a lock

Hibernation (part 06) already moved every space’s durable state into the bucket, and made the per-space lease the single arbiter of who may write it. That does most of the multi-box work by itself: any box could wake any space, and the lease already prevents two from doing so at once. What remains is efficiency and manners — spaces should not ping-pong between boxes, and a request that lands on the “wrong” box should still be answered. The mesh adds exactly three small mechanisms on top of the bucket:

  1. Membership: each box periodically writes a tiny heartbeat object; listing them yields the set of live boxes.
  2. Placement: a deterministic hashing rule (“rendezvous hashing”) maps each hostname to one preferred owner among the live boxes.
  3. Forwarding: a box that receives a request for a space it does not own transparently proxies the request to the owner.
In plain words

The boxes never talk to each other to decide who does what. Each box signs a guest book in the shared storage room every fifteen seconds, everyone applies the same arithmetic to the guest book to work out who should host which space, and the storage room’s key hooks (the leases) make it impossible for two boxes to act on a disagreement. A box that stops signing the book is presumed gone within forty-five seconds, and its spaces re-home themselves automatically.

Enabling the mesh on a box

Three environment variables in /data/router.env, on top of an already-configured hibernation bucket:

LCHAT_MESH_SELF — the master switch and the box’s identity
A URL with a scheme (https or http) and a host, no path. Its presence turns the mesh on; absent means a single box and zero mesh code. It must be reachable by the other boxes (it is what they proxy to), which may be a private network address (http://10.0.0.5) or a public one.
LCHAT_MESH_DOMAINS — the scope, and it is required
The router refuses to start with LCHAT_MESH_SELF set but no domains. Only hostnames equal to or directly under a listed domain participate in mesh placement; every other host is always served locally, exactly as without a mesh. Two reasons for the scoping, both practical: a routing entry for a host only exists on the boxes it has been replicated to (see route replication), so placement must be confined to hosts every member can serve; and every mesh member must be able to obtain a TLS certificate for any host that might land on it, which per-host issuance cannot promise (it ties a certificate to whichever box the public DNS points at) but a shared wildcard certificate can. In practice: mesh domains should be wildcard-certificate domains.
LCHAT_MESH_TOKEN — the shared secret, and it is also required
The router refuses to start with LCHAT_MESH_SELF set but no token. A bearer secret, identical across every mesh member, that authenticates the route-replication endpoint (below) — without it, any box that could reach another's mesh port could hand it arbitrary routing entries. Generate it once and copy it to every box’s /data/router.env; it is not derived from anything else and is not printed anywhere at runtime.

Membership: heartbeats in the bucket

Every fifteen seconds, each mesh box writes one object under <prefix>/mesh/members/ in the hibernation bucket and lists the others. The scheme has three properties worth appreciating:

  • The object’s key is the record — it is an encoding of the box’s own LCHAT_MESH_SELF URL — so a single cheap LIST operation yields the entire membership, with no per-member reads.
  • Liveness is judged by the bucket’s clock, not the boxes’ clocks: freshness is the object’s last-modified time as recorded by S3 itself, so a box with a wrong clock cannot poison anyone’s view. A heartbeat older than forty-five seconds (three missed beats) marks its box dead.
  • Failures are soft. A box that fails to write its beat is simply taken over by its peers (the lease arbitrates any overlap); a box that fails to read keeps its last known view and at worst misroutes — which, again, the lease absorbs. Neither failure is fatal, and both are logged.

Placement: rendezvous hashing, and why the lease still decides

For each in-scope hostname, every box independently computes a score for every live member (a hash of the pair “this box, this host”) and the highest score wins ownership. This is rendezvous hashing, and its virtue is minimal disruption: because each (box, host) pair’s score never changes, adding or removing one box only reassigns the hosts whose winning box appeared or vanished — roughly 1/N of them — while everything else stays put. There is no table to distribute, no leader to elect: ownership is arithmetic on the guest book.

But membership views can momentarily disagree (one box has seen a fresh heartbeat the other has not). The design’s answer is that hashing is only a hint, and the lease is the truth: a box that wrongly believes it owns a space simply loses the lease race at wake time, loudly, and the requester retries onto the right answer. Transient disagreement structurally cannot corrupt anything — it can only cost a retry.

Two refinements keep behaviour calm: a space that is currently awake on some box stays there until its natural sleep, even if the hash now prefers another box — its state is already in the bucket, so “migration” is nothing more than waking on the new owner next time. And scheduled wakes fire only on the box that owns the host, so a due reminder wakes one box, not all of them.

Forwarding: wrong-box requests proxy to the owner, once

When a request arrives at a box that is not the computed owner, the box reverse-proxies the entire request to the owner’s LCHAT_MESH_SELF URL — preserving the original hostname, verifying the owner’s TLS certificate against the instance’s hostname (this is why mesh domains want the shared wildcard certificate), and pinning the underlying connection to the peer’s address. There is no certificate-verification bypass anywhere. A forwarded request is marked with an internal header (X-Lchat-Mesh-Forwarded, stripped from all outside traffic), and a marked request is never forwarded again: if two boxes disagree about ownership, the second box serves the request locally — a wrong-box wake the lease then resolves — instead of the pair bouncing the request forever.

Route replication: every box learns about every host

The router looks a request’s hostname up in its own local registry before any mesh logic runs at all, and answers 404 immediately on a miss. So placement (hashing, forwarding) is only half the story: a box can only ever serve — or even correctly forward — a host it has a registry row for. Two mechanisms keep every mesh member’s registry in sync, both authenticated with LCHAT_MESH_TOKEN:

  • Automatic fan-out on creation. The moment a new in-scope hibernating host is registered anywhere, that box replicates the row to every currently live mesh peer (found the same way placement does — the membership heartbeats). Only a narrow, purpose-built payload crosses the wire — host, kind, version, hibernate flag, and a handful of routing-relevant fields — never the receiving box’s own socket path or database path, which each box always derives locally. A peer that is down at that moment simply misses the row; fan-out is best-effort, logged, and never blocks the create.
  • lchat mesh-seed <peer-url> — the one-time catch-up. Run on an existing box and pointed at a newly joined one, this bulk-replicates every in-scope hibernating route the existing box already knows about, in one pass. This is the command that makes joining a box a single step instead of hand-registering every host (see adding a box, below).
Why replicate rows instead of just answering for any host

The alternative — a box that does not recognise a hostname asking its peers before giving up — would need a new box-to-box RPC, adding a network dependency the S3-only design deliberately avoids everywhere else. Replicating the (small, non-secret) routing metadata ahead of time keeps every box able to answer a hit or a genuine 404 from its own disk, no peer round-trip required.

A replicated route still needs materializing before its first wake

A route reaching a box through replication (or mesh-seed) is a registry row only — unlike add-instance on the box that originates it, replication never stages a binary or creates a live symlink for that host. The router process is sandboxed (ProtectSystem=strict) and cannot write to /opt/lchat itself, so its wake path materializes a first-time route by asking the unsandboxed control daemon to do it instead. This needs the router unit’s LoadCredential=control-token:/etc/lchat/control.token drop-in (deploy/systemd/lchat-router-control-token.conf) — without it, the router logs a warning at startup and a replicated route’s first wake on that box fails outright (a live production incident the first time this was tried; see docs/planet-scale.md’s decision log for the full account). Every mesh member needs this drop-in installed.

Adding, draining, and losing a box

Adding a box to the mesh

  1. Provision the box normally.

    Install the binary and run lchat bootstrap, exactly as in part 04. Give it the same hibernation bucket, prefix, and credentials as its peers in /data/router.env — the shared bucket is the entire coordination fabric.

  2. Give it the shared wildcard certificate capability.

    Configure LCHAT_WILDCARD_DOMAINS (and the DNS API token) for the mesh domains, so this box can present valid certificates for any space that lands on it.

  3. Install the control-token drop-in.

    Copy deploy/systemd/lchat-router-control-token.conf in alongside lchat-router.service so the router can materialize a replicated route’s first wake through controld — see the warning above. Skipping this is the single most common way a fresh mesh box looks fine (heartbeats, membership) until the first real request lands on it.

  4. Set its mesh identity and restart the router.

    Within one heartbeat interval the peers see the new member.

  5. Seed its registry from an existing box.

    Bulk-replicates every in-scope hibernating route in one pass, so the new box can immediately answer for (or correctly proxy) any of them. After this, roughly 1/N of the in-scope spaces re-home to the new box at their next sleep/wake boundary — nothing needs announcing beyond this one command; the guest book and the registry rows do the rest.

Draining a box (deploys, maintenance, scale-down)

Stopping the router is the drain: its shutdown sequence deletes this box’s heartbeat first — so peers take ownership immediately — and then checkpoints every awake space to the bucket and releases its leases as usual. Deploying a new router build, retiring a box, and reacting to failure are deliberately the same code path.

The recovery numbers, verified by live tests in both directions: after a graceful drain, the survivor is serving the drained box’s spaces about two seconds later; after a hard kill (no drain, heartbeat left to rot), within about the lease lifetime — thirty seconds when those tests ran, fifteen minutes since #836 stretched the lease to tame its S3 request bill — with data intact in both cases (up to the space’s replication class’s window), because writes had been syncing to the bucket all along.

The honest limits: what stands between here and a true multi-box fleet

Two gaps remain. Neither is speculative — each corresponds to a concrete piece of code that is reserved or not yet written. (Two things this section used to list — registry replication and fleet-wide status aggregation — have since shipped and are covered above and in part 08.)

  • partially built Route mutations after creation do not replicate. The initial fan-out (above) and mesh-seed cover creation and joining, but a later rename, removal, or hibernation-flag flip on one box is not currently propagated to its peers — those must still be repeated by hand on every mesh member. Fine for a pool of auto-provisioned, rarely-renamed spaces; worth closing before mesh placement carries more than that.
  • partially built The certificate cache is per box. The wildcard certificate makes mesh domains workable (each box independently obtains its own copy), but per-host certificates for domains outside the wildcard cannot follow a space between boxes; the S3-backed shared cache that would fix this is reserved (LCHAT_CERT_CACHE=s3) and deliberately refuses to start. See part 02.
Designed, not built: what the mesh grows into

The written design (docs/planet-scale.md) carries the mesh further than any current code: per-space geographic placement using the registry’s home_region field (relocating a space between regions while it sleeps, by copying its bucket prefix and flipping the field); an autoscaling group of boxes behind a network load balancer, sized on warm-set memory; and a co-operative “provider” seam in which spaces could be delegated to operators other than the product’s own. Each of these has a schema field or an interface already cut for it, and none has operable behaviour today.
