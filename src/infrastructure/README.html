Maintaining this guide

The guide is nine plain HTML pages, one shared stylesheet, and one shared script — no build step, no dependencies, nothing to install. Open any page in a browser straight from the repository, or serve the directory statically. This page explains how to keep it accurate and how to extend it.

How the directory is put together

FileRole
index.html … reference.htmlThe nine content pages. Each is self-contained semantic HTML; nothing is generated.
guide.cssThe single stylesheet. Colors, typography, panels, badges, tables, diagrams — edit here and every page follows. Light and dark themes are both defined at the top.
guide.jsThe single script, and the guide’s one source of structure: the PAGES list (which drives the sidebar and the previous/next links on every page) and the shared “you are here” system-map diagram. All injected markup is built from static in-code strings only, per the repository rule about innerHTML.

Editing an existing page

  • The code is the source of truth. Before changing a factual claim, check it against the code (control.go, router.go, activator.go, mesh.go, controld.go, cloud.go, reaper.go, deploy/systemd/) or the design documents (docs/planet-scale.md, docs/instance-router.md). Where this guide and the code disagree, fix the guide.
  • Keep the register. The guide deliberately avoids contractions, avoids unexplained jargon, spells out commands in full, and repeats key ideas where they matter. Match that when adding text.
  • Keep the status badges honest. Anything not operable today must carry partially built or designed, not built, and aspirational behaviour belongs in a dashed .roadmap panel, never in running prose. When a roadmap item ships, promote its badge and move the text out of the panel in the same change.
  • Use the shared components rather than inventing new markup: figure.cmd for labelled command blocks, ol.procedure for step-by-step tasks, .plain / .warn / .why panels for recaps, cautions, and rationale, and .status badges. Examples of each are on every page.

Adding a page

  1. Copy an existing page as the template.

    Keep the <head>, the .frame/nav.side/main.content skeleton, the header block, and the div.sysmap. Set data-page on <body> to the new file name.

  2. Add one entry to the PAGES list in guide.js, in reading order.

    That single edit updates every page’s sidebar and previous/next links. Renumber the num fields and each page’s eyebrow (“Part NN of NN”) if the new page changes the ordering.

  3. Choose the system-map highlight.

    Set data-highlight on the div.sysmap to the node keys the page covers (members router instances services control updater bucket backup peer, space-separated). If the page introduces a genuinely new component, add a node for it to sysmapSVG in guide.js so the map stays a true whole-system picture.

Checking your changes

Eyeball both color schemes (the pages follow the operating system’s light/dark preference), confirm the sidebar highlights the page you are on, and confirm the previous/next links at the bottom of each page still make sense.

Authorship marking

Per the repository’s authorship convention (CLAUDE.md), these pages were written by an LLM and carry the visible 🤖 marker in each page’s eyebrow. A human who substantially rewrites a page should remove the marker from that page (or add a person marker to certify it); an LLM editing further keeps it.

Publishing to home.elevenmessenger.com and elevenmessenger.com

The guide is reachable in two places, and this directory is the only source for both — nothing is ever hand-copied or edited in place at the destination:

  • home.elevenmessenger.com/infrastructure/ — served straight from disk by the home server (home/main.go’s serveInfrastructure). A deploy step scp -rs this directory onto the box and sets LCHAT_HOME_INFRASTRUCTURE_DIR; see home/README.md § Optional: serving the planet-scale guide.
  • elevenmessenger.com/infrastructure/ — mirrored automatically into the public elevenmessenger/website repo (a separate, Eleventy-built static site) by .github/workflows/sync-guide-to-website.yml: any push to main that touches this directory copies it into that repo’s src/infrastructure/ (Eleventy passthrough-copies it unmodified — see its .eleventy.js) and pushes, which triggers that repo’s own existing deploy workflow. No manual step, and no separate copy of the guide to keep in sync by hand.
One-time setup: the website repo token

The sync workflow needs write access to the public elevenmessenger/website repo, granted via a fine-grained GitHub personal access token stored as a secret in this repo (messenger), named WEBSITE_REPO_TOKEN. To create it:

  1. Mint the token.

    GitHub → profile photo → Settings → Developer settings → Personal access tokens → Fine-grained tokens → Generate new token. Set Resource owner to the elevenmessenger organization, Repository access to Only select repositories → elevenmessenger/website (and no other repository), and under Permissions → Repository permissions set Contents to Read and write — leave every other permission at its default (No access). Give it an expiration and a name that says what it’s for (e.g. “messenger guide sync”).

  2. Add it as a secret on the messenger repo.

    On github.com/elevenmessenger/messenger: Settings → Secrets and variables → Actions → New repository secret. Name: WEBSITE_REPO_TOKEN. Value: the token from step 1.

  3. Run the sync once by hand.

    Actions tab → Sync planet-scale guide to the website → Run workflow (it also has a workflow_dispatch trigger for exactly this) — seeds website’s src/infrastructure/ for the first time. Every subsequent guide change on main syncs automatically.

Until the secret is set, the workflow runs, logs a warning, and does nothing else — it never fails loudly and never risks a partial or broken sync.

tools/sync-guide-to-website.sh <path-to-a-local-checkout-of-website> does the same copy-commit-push by hand, using your own git credentials instead of the token — useful when Actions can’t run (a billing/spending-limit block, say) or to try a guide change before merging it.
