Everything that runs on a box

This page is a complete census of the processes on a fleet box: what each one is, how it is started, how it is configured, how it is supervised, and how you check on it. It also maps the whole filesystem layout, because every process is defined by the files it reads and writes.

You are here: everything inside the “one box” zone. Each highlighted node is one or more of the processes catalogued below.

A one-minute systemd primer

Every long-running process on a box is supervised by systemd, the standard Linux service manager. Three ideas are enough to follow everything on this page:

  • A unit is a small text file in /etc/systemd/system/ describing one service: what command to run, as which user, with what restrictions, and what to do if it exits. All of this system’s unit files are written for you by lchat bootstrap (described in part 04); their sources are version-controlled in the repository under deploy/systemd/.
  • A template unit has an @ in its name and is instantiated per name after the @. So lchat@.service is a template, and lchat@chat-a.example.com.service is the concrete service for one instance — systemd substitutes the part after @ (written %i inside the unit file) wherever it appears. This one mechanism is how a single unit file serves any number of instances.
  • The verbs you will use constantly: systemctl status <unit> (is it running, since when, last log lines), systemctl restart <unit>, systemctl enable --now <unit> (start it and also start it at every boot), systemctl disable --now <unit> (stop it and stop starting it at boot), and journalctl -u <unit> (its full log).

Nearly every unit here sets Restart=always, meaning systemd restarts the process within a second or two if it ever exits. Crash recovery is therefore automatic; your job is to notice repeated crashing (see part 08), not to restart things by hand.

Seeing everything at once

These three views answer different questions, and it is worth internalising the difference. systemctl shows supervised processes. lchat instances shows the registry — the intended state of the box, including hosts that are asleep on purpose. ss shows reality at this instant — which sockets have a live listener. When the three disagree, the disagreement is itself the diagnostic: a ready registry row with no bound socket and no unit is a sleeping hibernator (normal); a running unit with no bound socket is a process failing to start (investigate its journal).

The filesystem layout: what lives where

Every process above is defined by files in three directories. This layout is worth memorising, because nearly every operational question (“where is this instance’s data?”, “where is its admin token?”, “which binary is it running?”) is answered by a path.

/opt/lchat/ — the software

PathContents
/opt/lchat/lchatThe control binary: what the router, control daemon, updater, and your own command-line invocations run.
/opt/lchat/versions/<ver>/lchatThe version store: every staged build, one directory per build identifier (a short git commit hash). Builds accumulate here and are garbage-collected when nothing references them. See part 04.
/opt/lchat/live/<host>One symbolic link per instance, pointing into the version store. This is the single most important path on the box: each instance’s unit runs /opt/lchat/live/<host>, so which build an instance runs is decided entirely by where its link points. Deploying is repointing a link; rollback is repointing it back.
/opt/lchat/incoming/A staging area for uploaded and downloaded binaries before they are verified and moved into the version store.
/opt/lchat/.deploy.lockA lock file that makes deploys, rollbacks, and updater passes take turns instead of interleaving. If two are attempted at once, the second fails immediately with a clear message.
/opt/lchat/manifest-stampThe timestamp of the newest release manifest the box has ever accepted — the auto-updater refuses anything older, so the release feed cannot be replayed backwards. See part 04.
/opt/lchat/home, /opt/lchat/pushThe sibling-service binaries.

/data/ — the state

PathContents
/data/<host>.db (+ -wal, -shm)One instance’s entire SQLite database. The -wal and -shm files are SQLite’s write-ahead log machinery and travel with the database. Contents are end-to-end ciphertext plus routing metadata.
/data/<host>.envThe instance’s environment file (mode 0600): its configuration as KEY=value lines, read by its unit at start. Critically, it holds LCHAT_ADMIN_TOKEN — the database stores only a hash of the admin token, so this file is the only recoverable copy. lchat admin-link <host> reads it for you.
/data/<host>-blobs/The instance’s encrypted attachment files (photos, videos), content-addressed, stored outside the database to keep the database small.
/data/<host>.vapidThe instance’s web-push signing key. Kept out of the database on purpose (so database replicas in object storage carry no private key). If you ever move an instance between boxes, copy this file too, or every browser push subscription silently breaks — see part 08.
/data/<host>-bot.env + /data/<host>-bot/The bot sidecar’s environment file and its persisted bot identities (session files). The existence of <host>-bot.env is, literally, what “this instance has a sidecar” means to the system.
/data/registry.dbThe registry — the box’s routing table and instance ledger.
/data/acme/The TLS certificate cache (private keys inside — see part 02).
/data/router.env, control.env, cloud.env, home.env, push.env, update.envPer-service environment files, one per unit, same KEY=value format. Each unit declares its own; most are optional.
/data/home.db, push.db, cloud.dbThe sibling services’ own databases.
/data/trash/<host>-<timestamp>/Archived data of deleted spaces. Deletion in this system archives, never destroys — see part 05.

/run/lchat/ — the sockets

A memory-backed directory (recreated at every boot) holding the unix sockets: <host>.sock for each awake instance, plus home.sock, push.sock, cloud.sock, and control.sock. Sockets are transient by nature; nothing here needs backing up.

Two more paths complete the picture: /etc/lchat/control.token (a root-only secret gating the control daemon — part 05) and /etc/litestream.yml (the fleet-backup configuration — part 08).

The instance process, in detail

Each chat space runs as one instance of the template unit lchat@.service. Reading its key lines teaches you most of the process model:

  • The host is the identity. The name after @ flows into every path: the binary link, the socket, the database, the environment file. Knowing a host, you can find everything about it.
  • Configuration is the environment file. The instance reads ordinary environment variables (LCHAT_ADMIN_TOKEN, LCHAT_PUSH_RELAY_URL, and so on); the unit points them at /data/<host>.env. Changing an instance’s configuration means editing that file and running systemctl restart lchat@<host>.
  • The version is the symlink. ExecStart runs /opt/lchat/live/<host>, so a deploy changes the link and restarts the unit — nothing about the unit itself changes. readlink /opt/lchat/live/<host> tells you exactly which build an instance runs.
  • Resource limits are per instance. MemoryMax=512M means one runaway instance is killed by the kernel before it can starve the others — isolation is enforced, not hoped for.
  • Graceful shutdown is built in. On being told to stop, the instance finishes in-flight requests for up to ten seconds and closes its WebSocket connections cleanly (clients auto-reconnect), then exits. Combined with the router’s thirteen-second dial retry, this is why restarts do not drop requests.

The security posture every unit shares

All the units are heavily sandboxed with systemd’s confinement features: they run as the unprivileged system user lchat, see a read-only filesystem except /data and /run/lchat, cannot gain new privileges, cannot load kernel modules, and are memory-capped. Two deliberate exceptions run as root: the control daemon and the auto-updater, because their whole job is to do the two privileged things (start and stop units via systemctl, and write the version store). Both are reachable only locally — the control daemon on a local socket gated by a root-only token, the updater only via its timer. The router is the one unit with a special capability (CAP_NET_BIND_SERVICE) so it can bind ports 80 and 443 without being root.

In plain words

Everything runs as a locked-down, unprivileged user except the two processes whose job is administration itself, and those two never talk to the internet. If an instance is somehow compromised, the sandbox means it cannot read other instances’ secrets, cannot touch the software store, and cannot escalate.

The sibling services: home and push

Two services deploy next to the messenger but are not chat instances, and the router treats them as plain service routes — names it forwards traffic to, whose lifecycle it does not manage:

  • The home server (lchat-home.service) is the account-recovery backup store: members’ encrypted key backups, gated by passkeys. Its one operational constraint is that its hostname must never change — the hostname is part of the passkey security model (the WebAuthn “relying party” origin), and changing it orphans every passkey. Its environment file /data/home.env is required (no leading - in the unit) because a misconfigured home server is worse than a stopped one.
  • The push relay (lchat-push.service) holds the one Apple push-notification signing key for the whole fleet, so that individual instances never do. Instances authenticate to it with per-instance revocable tokens. Its environment file /data/push.env is likewise required.

Both are registered with the router by lchat add-service <host> --socket <path>, which simply adds a service row to the registry after checking the backend answers. Deploying either is manual and rare: copy the new binary over the old one and systemctl restart the unit (part 04).

Bot sidecars: a member that runs on a server

An instance can have a bot sidecar: a process hosting the built-in helper personas (Claude and Eleven). Architecturally, a bot is an ordinary end-to-end encrypted member of the chat whose “device” happens to be a server process: it holds its own private keys (in the session files under /data/<host>-bot/) and connects to its instance as a WebSocket client, over the instance’s public hostname, exactly like a browser would. It serves no inbound traffic at all — no socket, no port, no route.

Why it is this way

The product’s core rule is that the server process never holds a usable decryption key. A bot must decrypt the threads it participates in, so its keys must live in a different process — one the operator can place anywhere (the same box, a different machine, behind a home router), because it only ever dials out. Compromising the server must not yield bot keys; this separation is what guarantees that.

Operationally: lchat add-bot <host> provisions the identities and writes /data/<host>-bot.env; the unit lchat-bot@<host>.service runs /opt/lchat/live/<host> bot run — note that it execs the same versioned symlink as its instance, so the bot always runs the same build as its app, by construction, and every deploy restarts it onto the matching version. Give it intelligence by adding ANTHROPIC_API_KEY=… to its environment file and restarting; without a key the bots still run with deterministic (non-AI) behaviour.

One subtlety worth knowing even if you never touch it: because the sidecar dials its instance’s public URL, the box must resolve its own hostnames to itself. Every hosted hostname is pinned to 127.0.0.1 in /etc/hosts, so on-box clients reach the local router directly instead of going out to the internet and back. Cloud images that regenerate /etc/hosts on boot have that behaviour disabled. If a sidecar logs connection failures after a migration, stale DNS or a missing hosts-pin is the first thing to check.

On a hibernating instance the sidecar unit is disabled and the activator co-spawns the sidecar only while the instance is awake — that story is told in part 06.

How to change any process’s configuration

The pattern is identical everywhere, and worth stating once as a rule: configuration lives in an environment file under /data/, and takes effect on restart.

  1. Edit the environment file for the process.

    For an instance: /data/<host>.env. For the router: /data/router.env. For a sidecar: /data/<host>-bot.env. And so on per the table above. Keep the file’s ownership and 0600 mode — it may hold secrets.

  2. Restart (or reload) the corresponding unit.

  3. Confirm the process came back healthy.

One exception

A hibernating instance has no unit to restart. Its environment file is read fresh by the activator at every wake, so an edit takes effect at the next wake with no action from you — or force the matter by making any request to the host. See part 06.
